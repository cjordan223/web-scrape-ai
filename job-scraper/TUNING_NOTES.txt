FILTER TUNING NOTES
===================
Last updated: 2026-03-15

PURPOSE
-------
Use this file as a quick operator guide for tuning scraper pass/reject behavior.
For full config definitions, see:
- job_scraper/config.default.yaml

HOW TO REVIEW CURRENT REJECTIONS
--------------------------------
1. Start dashboard backend:
   python dashboard/backend/server.py
2. Open:
   http://localhost:8899
3. Go to:
   Scraping -> Intake -> Rejected
4. Filter by stage and inspect rejection reasons.

Useful SQL:
  SELECT rejection_stage, rejection_reason, title, url
  FROM rejected
  ORDER BY created_at DESC
  LIMIT 200;

HIGH-IMPACT KNOBS
-----------------
filter.min_salary_k
  - Raises/lowers strictness for salary gating.

filter.require_remote
  - If true, explicit onsite/in-office postings are rejected.

filter.require_known_board
  - If true, unknown board domains are rejected.
  - Default: false (relaxed 2026-02-24).

filter.max_experience_years
  - Tunes experience matching strictness.

filter.seen_ttl_days
  - URLs older than this in seen_urls are re-eligible for processing.
  - Default: 14.
  - Use a positive integer day count. The current implementation does not treat `0` as a "disable TTL" switch, so do not set `0` expecting permanent dedup behavior.

filter.score_accept_threshold / score_reject_threshold
  - Controls pass/review/reject banding.

llm_review.enabled
  - Enables/disables LLM review stage.

llm_review.fail_open
  - If true, LLM outages AND unparseable responses do not block jobs.

TUNING WORKFLOW
---------------
1. Pick one stage causing excessive rejection.
2. Adjust the smallest possible config knob.
3. Run a scrape.
4. Compare:
   - accepted count
   - rejected by stage
   - quality of accepted jobs
5. Repeat.

OPERATING GUIDELINES
--------------------
- Prefer incremental changes over large threshold jumps.
- Keep hard blocks for clear policy violations.
- Use scoring + quarantine for uncertain cases.
- Re-check LLM review reasons if it becomes over-aggressive.

NOTES
-----
- URL canonicalization is critical for dedup quality.
- Missing JD text is handled as a softer signal than before.
- Hybrid handling and US-location policy are controlled by filter settings.
